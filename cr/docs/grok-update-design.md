# grok update — Design Doc

**Date**: 2026-06-04
**Author**: CodeRhapsody + Bill Cox

## Problem

.grok files are the primary design artifact in GDD, but they have two maintenance problems:

1. **Creation friction**: Writing a .grok file from scratch for an existing Go package is tedious. You have to manually transcribe every exported type, field, method, and function.
2. **Staleness**: As code evolves, the function signature index drifts. Updating it requires manually re-reading Go source and editing the .grok file.

The verifier catches drift but doesn't fix it. We need a tool that **generates and updates** .grok files from Go source automatically.

## Design

### Three-Zone .grok File

A .grok file has three distinct zones, separated by marker comments:

```
// mypackage.grok — the Grok package's own understanding

grok MyPackage {
  why: "..."

  doc "Architecture": """
    ...
  """

  // Structural declarations (structs, enums, interfaces, classes)
  struct Foo { ... }
  interface Bar { ... }

  source: ["foo.go", "bar.go"]
}

// --- index ---
//
// Auto-generated function/method index with signatures in Go syntax.
// DO NOT EDIT below this line — regenerated by `grok update`.
//
// file: foo.go
//
// NewFoo creates a Foo with default settings.
func NewFoo(name string) *Foo                                         // :23
//
// Process runs the main processing loop.
func (f *Foo) Process(ctx context.Context) error                      // :45
//
// helper trims whitespace and normalizes.
func helper(s string) string                                          // :102
//
// file: bar.go
//
// Validate checks structural integrity.
func Validate(f *Foo) (bool, error)                                   // :15

// --- dependencies ---
//
// Auto-generated internal package dependencies.
// DO NOT EDIT below this line — regenerated by `grok update`.
//
// pkg/ast
// pkg/parser
```

### Zone Definitions

**Zone 1 — Human-curated (Grok syntax)**: Everything from the start of the file through the end of the last `grok { }` block. Contains `why:`, `doc` blocks, structural type declarations (structs, enums, interfaces, classes, functions, relations), annotations, and `source:` directives. The human (or AI with human review) maintains this zone, but current `grok update` parses and re-emits it so it can auto-add missing exported symbols. With `--prune`, it can also remove stale declarations. Review Zone 1 diffs after running update.

**Zone 2 — Function/method index (Go syntax)**: Starts at `// --- index ---` marker. Contains every function and method in the package, organized by source file, with:
- The first line of the Go doc comment (or `//` comment immediately preceding the func)
- The full Go function signature (receiver, name, params, returns)
- Line number as a trailing `// :NNN` comment
- Both exported and unexported functions included
- Blank line between entries for readability

This zone is **completely regenerated** on every `grok update` run.

**Zone 3 — Dependencies (auto-generated)**: Starts at `// --- dependencies ---` marker. Lists internal (project-only) package imports, deduplicated across all source files. External stdlib and third-party imports are excluded. This zone is **completely regenerated** on every `grok update` run.

### Commands

#### `grok update <file.grok>`

Reads an existing .grok file, updates Zone 1 structural declarations from Go source, then regenerates Zones 2 and 3 from the Go source files listed in `source:` annotations.

**Algorithm:**
1. Read the .grok file as raw text
2. Find the `// --- index ---` marker (or end of file if absent)
3. Parse Zone 1 to extract `source:` file paths from grok blocks
4. Resolve source paths relative to the .grok file's directory
5. Extract exported Go symbols from the listed source files
6. For each source file, use `go/parser` + `go/ast` to extract:
   - All function declarations (top-level and methods)
   - The first line of each function's doc comment
   - The exact source line number
   - The function signature as written in Go source
7. Organize by file, sort by line number within each file
8. For dependencies: collect all import paths across source files, filter to project-internal only (same module prefix from go.mod)
9. Re-emit updated Zone 1, then append Zone 2 (index) and Zone 3 (dependencies)

**Key detail**: Zone 1 is parsed and re-emitted by the formatter. Comments are retained, but formatting and declaration order may change. Newly exported Go symbols are added to Zone 1; `--prune` also removes stale exported declarations.

#### `grok gen <package-dir>`

Creates a new .grok file from scratch for a Go package.

**Algorithm:**
1. Find all `.go` files in the directory (excluding `_test.go`)
2. Parse all files with `go/ast`
3. Generate Zone 1:
   - `grok PackageName {` block
   - `why: ""` placeholder
   - For each exported type: generate Grok struct/interface declaration
   - `source: [list of .go files]`
   - `}`
4. Generate Zone 2: function index (same as `grok update`)
5. Generate Zone 3: dependencies (same as `grok update`)
6. Write to `<package-dir>/<package-name>.grok`

The generated Zone 1 has structural declarations but empty `why:` and no `doc` blocks — the human fills those in. This is the scaffolding.

#### `grok fmt <file.grok>`

Reformats Zone 1 of a .grok file using standard formatting rules and appends existing Zones 2 and 3 unchanged.

### Function Signature Extraction

We use `go/parser` and `go/ast` (the Go standard library) to extract signatures. This gives us:

- **Receiver**: from `FuncDecl.Recv` — format as `(name Type)` or `(name *Type)`
- **Name**: from `FuncDecl.Name`
- **Params**: from `FuncDecl.Type.Params` — reconstruct from Go AST field list
- **Returns**: from `FuncDecl.Type.Results` — single value or tuple
- **Doc comment**: from `FuncDecl.Doc` — take first line, trim `//` prefix
- **Line number**: from `FuncDecl.Pos()` via `token.FileSet`

We emit the signature as close to the original Go source as possible. No Grok translation — this is a navigation index in the target language.

### Dependency Extraction

For each source file:
1. Walk `ast.File.Imports`
2. Extract import path strings
3. Read `go.mod` from the project root to get the module path
4. Filter: keep only imports that start with the module path
5. Strip the module path prefix to get relative package paths
6. Deduplicate across all source files
7. Sort alphabetically

### File Discovery

Source files come from two places:
- **`grok update`**: reads `source:` annotations from the existing .grok file
- **`grok gen`**: scans the directory for all `.go` files, excluding `*_test.go`

For `grok update`, if the `source:` annotation is missing or empty, fall back to scanning the .grok file's directory for `.go` files.

### Edge Cases

- **No source annotation**: Fall back to directory scan
- **Source file doesn't exist**: Warn and skip
- **No functions in a file**: Emit the `// file:` header with no entries
- **No doc comment**: Emit the signature with no comment line above it
- **Multi-line doc comments**: Take only the first line
- **Multiple grok blocks**: Collect `source:` from all blocks, deduplicate
- **Init functions**: Include `func init()` in the index
- **Generated files**: Include them (the human can add exclusion later if needed)
- **Build-tagged files**: Include all files regardless of build tags for completeness

## Implementation Status

The original phased plan has been implemented under both standalone commands and the unified `grok` CLI.

Available commands:

```
grok verify <file.grok>
grok update [--prune] <file.grok>
grok gen <package-dir>
grok fmt <file.grok>
grok compile <file.gk>
```

The standalone `grok-update`, `grok-gen`, `grok-verify`, and `grok-compile` commands are still present.

## Non-Goals

- **Semantic validation of Zone 1 docs**: `grok update` only syncs structural declarations and generated indexes. It does not prove narrative docs, invariants, relations, or pre/postconditions.
- **Semantic analysis**: The function index is purely syntactic — no type checking, no Grok type translation.
- **Cross-language support**: This design targets Go. Other languages (Python, Rust) would need their own extractors but the .grok file format stays the same.
- **IDE integration**: Future work. The function index with line numbers makes this possible.
- **Full comment fidelity**: Comments are retained by the parser/formatter, but Zone 1 is re-emitted in canonical format rather than preserved byte-for-byte.

## Testing Strategy

- Unit tests for signature extraction (various Go func patterns: methods, generics, variadic, multi-return)
- Unit tests for dependency filtering (module path matching)
- Integration test: run `grok update` on existing .grok files in this project, verify expected Zone 1 additions/pruning and Zone 2 output
- Golden file tests: known .go input → expected .grok output

## Open Questions

1. **Should the index include struct/interface type declarations too?** Currently only functions. Types are in Zone 1 (Grok syntax). But some types are missing from Zone 1 — the completeness check would catch this, but having them in the index too would be redundant.

2. **Should unexported functions be included?** Yes for now — they're useful for navigation. The `// file:` headers already group by file which provides organizational context. Could add a flag `--exported-only` later.

3. **Should we include function body line count?** E.g. `// :342 (45 lines)`. This would help estimate complexity at a glance. Low cost to add.
